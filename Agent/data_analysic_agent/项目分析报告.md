# 数据分析 Agent 项目分析报告

---

## 一、项目概览

本项目是一个**基于 LLM 驱动的自动化数据分析 Agent**，核心理念是：用户只需提供一份 CSV/Excel 数据文件，Agent 即可自动完成从**数据探索 → 维度规划 → 维度分析 → 图表生成 → 结论提炼 → 报告输出**的全链路数据分析任务，最终生成包含图表和分析结论的 Markdown 报告。

## 二、完整流程详解

整个 Agent 的执行流程可分为 **6 个阶段**，按顺序串联执行：

### 阶段 1：数据读取与预览

**入口**：[main.py](file:///f:/prompt-engineering/Agent/data_analysic_agent/main.py) → `DataExplore(data_path, query).run()`

**数据流**：

```
CSV/Excel 文件 → read_data() → pandas DataFrame → data_random_sample() → Markdown 表格字符串
```

**细节**：

1. `read_data()` 支持两种格式：
   - CSV：先尝试 UTF-8 编码，失败后回退 GBK
   - Excel：先尝试 openpyxl 引擎，失败后回退 xlrd 引擎
2. 读取后对列名做清洗（去除前后空格、换行符），删除全空行
3. `data_random_sample()` 从全量数据中**随机抽取 50 条**，转换为 Markdown 表格格式，作为后续 LLM 的输入上下文
4. 单元格内容超过 30 字符会被截断，列数超过 30 列只保留前 30 列

**设计意图**：随机抽样 50 条是为了在保留数据分布特征的同时控制 Token 消耗，避免将全量数据发送给 LLM。

### 阶段 2：分析维度规划（LLM 驱动，当前硬编码）

**核心函数**：`general_analysis_dimension()` in [data_explore.py](file:///f:/prompt-engineering/Agent/data_analysic_agent/workflow/data_explore.py)

**数据流**：

```
Markdown 数据样本 + 用户分析建议 → analysis_dimension_prompt → LLM (Qwen3-32B) → 结构化配置 JSON → parse_profiles_data_safe() → profiles_list
```

**细节**：

1. 将阶段 1 生成的 Markdown 表格 + 用户 query 填入 `analysis_dimension_prompt`
2. Prompt 要求 LLM 以 Python 变量赋值格式（`profiles1 = {...}`, `profiles2 = {...}`）输出分析维度配置
3. 每个配置项包含：
   - `analysis_id` / `analysis_name`：维度标识与名称
   - `preprocessing`：预处理配置（是否 drop_na、日期列、日期格式）
   - `dimensions`：维度分析配置列表（维度列、指标列、聚合方法、排序规则）
   - `time_series_analysis`：时间序列分析配置
   - `correlation_analysis`：相关性分析配置
4. 使用正则 `profiles\d+ = ({.*?})\n\n` 提取 LLM 输出中的配置字典
5. 使用 `ast.literal_eval()` 安全解析（而非 `eval()`）

> ⚠️ **当前状态**：该功能已被注释，`profiles_list` 被硬编码为大学专业数据的 4 个分析维度。说明 LLM 生成的维度配置质量不稳定，尚未达到生产可用。

### 阶段 3：维度数据分析（确定性计算引擎）

**核心类**：`DataAnalyzer` in [analysis_dimension.py](file:///f:/prompt-engineering/Agent/data_analysic_agent/workflow/analysis_dimension.py)

**数据流**：

```
profiles_list 中的每个 profile + 原始 DataFrame → DataAnalyzer → 预处理 → 维度聚合 → 时间序列分析 → 相关性分析 → 结构化结果（含 Markdown 表格）
```

**细节**：

1. **配置解析**：`_parse_config()` 将 JSON/dict 配置转化为 dataclass 对象（`AnalysisConfig`, `DimensionConfig`, `TimeSeriesConfig`, `CorrelationConfig` 等）
2. **智能列名匹配**：`_auto_match_columns()` 当 LLM 生成的列名与实际列名不完全一致时，通过三级匹配策略修正：
   - 精确匹配（忽略大小写）
   - 包含匹配（互相包含）
   - 相似度匹配（共同字符比例 > 0.5）
3. **数据预处理** `preprocess_data()`：
   - 禁用全局 drop_na（避免误删数据），只对特定列处理
   - 日期列转换为 `pd.to_datetime`
   - 指标列尝试转为数值类型（逐值检测，全部为数字才转换），缺失值填充为 0
   - 维度列缺失值填充为 "未知"
4. **维度聚合** `analyze_dimension()`：
   - 基于 `dimension_column` 分组
   - 使用 `pd.pivot_table` 执行聚合（支持 count, sum, mean, nunique, value_counts 等）
   - 若 `value_counts` 失败，自动回退为 `count`
   - 若 pivot_table 完全失败，降级为手动遍历聚合
   - 支持排序和 Top N 截断
   - 输出 Markdown 表格
5. **时间序列分析** `analyze_time_series()`：按指定频率重采样（resample）聚合
6. **相关性分析** `analyze_correlation()`：计算 Pearson 相关系数矩阵
7. **最终输出格式**：
```python
{
    "analysis_id": "...",
    "analysis_name": "...",
    "timestamp": "...",
    "dimensions": [{"name": "...", "markdown": "| ... |"}],
    "time_series": {...} or None,
    "correlation": {...} or None
}
```

### 阶段 4：图表生成（LLM 生成代码 + 执行）

**核心类**：`ImageCreate` in [img_create.py](file:///f:/prompt-engineering/Agent/data_analysic_agent/workflow/img_create.py)

**数据流**：

```
阶段 3 的分析结果 → img_create_prompt → LLM → PyECharts 代码 → parse_code() → exec() → PNG 图表文件
```

**细节**：

1. **异步并发**：使用 `asyncio.gather()` 并发执行所有维度的图表生成任务
2. **Prompt 设计**：`img_create_prompt` 将维度分析结果（Markdown 表格）作为输入，要求 LLM 选择合适的图表类型并生成 PyECharts 代码
3. **代码提取**：`parse_code()` 使用正则从 LLM 输出中提取 ` ```python ... ``` ` 代码块
4. **自动补全保存逻辑**：在代码末尾追加 `make_snapshot()` 调用，将 PyECharts 渲染为 PNG
5. **自动纠错**：若 `exec(code)` 执行失败，将报错信息 + 原代码发给 LLM 要求修复，**最多重试 1 次**
6. **输出**：每个维度生成一张 PNG 图表，保存在 `save_chart/` 目录

> ⚠️ **关键风险**：使用 `exec()` 直接执行 LLM 生成的代码，存在严重安全隐患。

### 阶段 5：维度结论生成（LLM 驱动）

**核心函数**：`get_dimension_conclusion()` in [data_explore.py](file:///f:/prompt-engineering/Agent/data_analysic_agent/workflow/data_explore.py)

**数据流**：

```
阶段 4 的 charts_results（含 Markdown 分析结果 + 图表路径）→ analysis_conclusion_prompt → LLM → title + txt
```

**细节**：

1. **异步并发**：同样使用 `asyncio.gather()` 并发生成所有维度的结论
2. **Prompt 设计**：`analysis_conclusion_prompt` 要求 LLM：
   - 提取表格标题赋给 `title` 变量
   - 生成专业的数据分析结论赋给 `txt` 变量
   - 对关键数据进行 HTML 渲染标记（颜色、加粗、高亮等）
3. **解析**：`parse_conclusion()` 使用正则提取 `title = "..."` 和 `txt = """..."""`
4. 将 title、txt、save_path 合并到每个维度的结果字典中

### 阶段 6：报告生成与输出

**核心类**：`GenerateReport` in [generate_report.py](file:///f:/prompt-engineering/Agent/data_analysic_agent/workflow/generate_report.py)

**数据流**：

```
dimension_conclusion → conclusion_concat() → 各维度结论拼接（含图片引用）
                   → summary() → LLM 总结 → 综合结论
                   → 最终 Markdown 报告 → save_markdown()
```

**细节**：

1. **结论拼接** `conclusion_concat()`：
   - 每个维度格式化为：`# 标题\n![图片](相对路径)\n- 结论文本`
   - 图片路径特殊处理：转换为 `../save_chart/filename` 相对路径（Markdown 不支持绝对路径）
2. **综合总结** `summary()`：
   - 将所有维度的 Markdown 表格数据拼接送入 `summary_prompt`
   - LLM 生成 500 字以内的综合分析结论
   - 要求使用 HTML 样式标注重要数据
3. **最终报告格式**：
   ```markdown
   # 数据报告
   ----
   [各维度结论 + 图表]
   ----
   # 总结
   [LLM 综合结论]
   ```
4. **输出**：保存为 `.md` 文件到 `output_report/` 目录（PDF 功能已注释）

## 四、设计思路分析

### 4.1 LLM + 确定性引擎的混合架构

项目采用了 **LLM 负责理解与生成、确定性代码负责计算与执行** 的混合架构：

| 环节 | 执行者 | 原因 |
|------|--------|------|
| 维度规划 | LLM | 需要理解数据语义和用户意图 |
| 数据聚合计算 | Pandas (确定性) | 需要精确的数值计算，LLM 不可靠 |
| 图表代码生成 | LLM | 需要根据数据特征选择图表类型和样式 |
| 图表渲染 | PyECharts (确定性) | 需要可靠的渲染引擎 |
| 结论生成 | LLM | 需要自然语言理解和专业表达 |
| 报告拼接 | Python (确定性) | 结构化拼接，不需要 LLM |

**核心思想**：将 LLM 用在需要"理解"和"创作"的环节，将确定性计算用在需要"精确"的环节，避免让 LLM 直接做数学运算。

### 4.2 配置驱动的分析引擎

阶段 3 的 `DataAnalyzer` 采用了**配置驱动**的设计：

- 输入是一份结构化的 JSON 配置（由 LLM 生成或人工编写）
- 引擎根据配置执行固定的分析流程
- 配置格式定义了完整的能力边界：维度聚合、时间序列、相关性分析

这种设计使得分析逻辑可复用、可测试，且与 LLM 解耦。

### 4.3 异步并发优化

在图表生成和结论生成阶段使用 `asyncio.gather()` 并发执行，提高处理效率。但由于底层 `modelscope_api()` 实际是同步调用（OpenAI SDK 的流式调用但无 `await`），**异步并未真正生效**，实际是串行执行。

### 4.4 容错机制

- 图表代码执行失败后自动重试 1 次（将错误信息反馈给 LLM 纠错）
- 维度聚合 `pivot_table` 失败后降级为手动遍历聚合
- 数据读取支持多编码/多引擎回退
- `value_counts` 不被 pivot_table 支持时自动替换为 `count`

### 4.5 智能列名匹配

当 LLM 生成的配置中列名与实际数据列名不完全一致时，通过三级匹配策略自动修正，增强了 LLM 输出的鲁棒性。

## 五、优势分析

### 1. 自动化程度高
用户只需提供数据文件，Agent 自动完成全链路分析，无需手动编写分析代码。对于非技术人员，大大降低了数据分析的门槛。

### 2. LLM + 确定性引擎的合理分工
将精确计算（数据聚合、排序）交给 Pandas，将理解和创作（维度规划、结论生成）交给 LLM，避免了 LLM 直接做数值计算容易出错的问题。

### 3. 配置驱动的分析引擎
`DataAnalyzer` 通过 JSON 配置驱动，使得分析逻辑可复用、可扩展。新增分析类型只需扩展配置格式和引擎逻辑，不需要修改上游流程。

### 4. 多层容错设计
从数据读取（多编码回退）到维度聚合（降级策略）到图表生成（LLM 纠错重试），每一层都有容错机制，提高了端到端的成功率。

### 5. 智能列名匹配
LLM 生成的列名可能与实际不一致，三级匹配策略有效缓解了这个问题，增强了系统的鲁棒性。

### 6. 输出质量较高
最终报告包含数据表格、可视化图表、专业结论和综合总结，信息密度高，对决策有参考价值。

## 六、劣势与风险分析

### 1. 🔴 exec() 执行 LLM 代码的严重安全风险
**这是项目最大的安全隐患。** `exec(code)` 直接执行 LLM 生成的代码，没有任何沙箱隔离。恶意或错误的代码可能：
- 读取/删除本地文件
- 执行系统命令
- 访问网络
- 造成无限循环或内存溢出

**建议**：使用 Docker 容器或 RestrictedPython 等沙箱环境执行 LLM 生成的代码。

### 2. 🔴 API Key 硬编码
`modelscope_api.py` 和 `token_calc.py` 中 API Key 直接硬编码在源码中，存在泄露风险。应使用环境变量或配置文件管理。

### 3. 🟡 异步并发名不副实
`ImageCreate.run()` 和 `get_dimension_conclusion()` 使用了 `asyncio.gather()` 并发，但底层 `modelscope_api()` 是同步阻塞调用。`async def run_task()` 中直接调用同步函数，并不会真正并发执行。需要使用 `asyncio.to_thread()` 或异步 HTTP 客户端才能真正并发。

### 4. 🟡 维度规划功能尚未可用
LLM 生成维度配置的功能已被注释，使用硬编码替代。说明 LLM 输出的结构化配置质量不稳定，解析可靠性不足。这是整个 Agent 智能化程度的关键瓶颈。

### 5. 🟡 LLM 调用成本与延迟
整个流程需要多次调用 LLM：
- 维度规划（1次）
- 图表代码生成（N次，N=维度数）
- 图表纠错（0~N次）
- 维度结论生成（N次）
- 综合总结（1次）

以 4 个维度为例，至少需要 10 次 LLM 调用。每次调用 Qwen3-32B 的流式推理，总延迟可能达到数分钟。

### 6. 🟡 数据抽样丢失信息
随机抽取 50 条数据发送给 LLM 做维度规划，可能遗漏数据的重要特征（如长尾分布、极端值、数据质量问题的具体表现）。

### 7. 🟡 正则解析 LLM 输出脆弱
- 维度配置使用 `profiles\d+ = ({.*?})\n\n` 正则提取
- 结论使用 `title = "(.*?)"` 和 `txt = """(.*?)"""` 正则提取

LLM 输出格式稍有偏差就会解析失败，且没有充分的 fallback 策略。

### 8. 🟡 仅支持单表分析
当前只能处理单个 CSV/Excel 文件，不支持多表关联分析。从 `required.txt` 中可以看到用户实际需求涉及多表（如学校表+专业表），但 Agent 无法处理。

### 9. 🟢 PDF 生成功能未启用
`MarkdownToPDF` 类已实现但被注释，说明 PDF 渲染效果可能不理想（中文字体、图片路径等问题）。

### 10. 🟢 错误处理粒度不够
维度分析出错时只记录日志并返回 error 字段，但上游流程没有对 error 做特殊处理，可能导致后续阶段（图表生成、结论生成）接收到不合法的输入。

## 七、改进建议

### 短期改进
1. **API Key 管理**：迁移到环境变量或 `.env` 文件
2. **真正的异步并发**：将 `modelscope_api()` 改为真正的异步调用（使用 `httpx.AsyncClient` 或 `asyncio.to_thread()`）
3. **LLM 输出解析增强**：使用 JSON Mode 或 Function Calling 替代正则解析，提高解析可靠性
4. **执行沙箱**：使用 Docker 或 RestrictedPython 隔离 LLM 生成的代码执行

### 中期改进
5. **维度规划质量提升**：引入 Few-shot 示例 + 输出校验 + 自动重试，使 LLM 维度规划真正可用
6. **多表关联分析**：支持用户上传多个数据文件，自动识别关联字段
7. **增量分析**：缓存中间结果，避免重复计算和重复 LLM 调用
8. **用户交互**：支持用户在维度规划后确认/调整维度，在报告生成后追问

### 长期改进
9. **Agent 架构升级**：引入 ReAct / Plan-and-Execute 框架，支持动态规划和工具调用
10. **评估体系**：建立分析质量评估基准，自动化测试端到端输出质量
11. **多模型路由**：简单任务用小模型（低成本），复杂任务用大模型（高质量）

---

## 八、总结

本项目是一个具有实际价值的 LLM 驱动数据分析 Agent 雏形，核心设计思路（LLM 负责理解与生成 + 确定性引擎负责计算）是正确的。项目已跑通了从数据到报告的完整链路，并在容错、列名匹配等方面做了细致的工程处理。

当前的主要瓶颈在于：**维度规划的 LLM 输出质量不稳定**（导致硬编码替代）、**异步并发未真正生效**、**exec() 存在安全风险**。解决这些问题后，项目可以演进为一个更成熟的数据分析 Agent 系统。
