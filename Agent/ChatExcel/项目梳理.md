# ChatExcel 项目梳理

## 1. 项目一句话定位

ChatExcel 是一个基于 FastAPI + React/Vite 的独立数据分析应用，用于上传 Excel/CSV 等表格文件后，通过大模型、DuckDB SQL 分析和 ReAct Agent 两种模式完成自然语言数据问答与可视化结果展示。

## 2. 这个代码仓解决什么问题

### 2.1 主要痛点

表格数据分析常见痛点包括：

1. **非技术用户不会写 SQL 或 Python**：业务人员面对 Excel、CSV 等文件时，往往只能手工筛选、透视或依赖开发人员。
2. **表结构不统一且语义不明确**：原始文件可能包含中文列名、多个 Sheet、重复列名、空列、日期/数字混合类型等问题，直接查询不稳定。
3. **多轮数据问答需要上下文**：用户往往会连续追问，例如先问汇总，再问某个维度拆分，需要保留会话历史和已上传文件状态。
4. **单表 SQL 分析和复杂分析场景不同**：简单问题适合模型生成 SQL 后由 DuckDB 执行；复杂问题可能需要多步推理、执行 Python 代码、查看中间结果。
5. **分析结果需要可视化和可追溯**：用户不仅需要最终答案，还需要看到 SQL、表格结果、图表以及 ReAct 的执行步骤。

### 2.2 项目的处理思路

项目提供两条主要分析链路：

1. **ChatExcel SQL 模式**
   - 用户上传 Excel/CSV 文件。
   - 后端将文件按会话写入上传目录，并将每个非空 Sheet 或 CSV 落入会话级 DuckDB。
   - 学习阶段读取表结构、样例数据和统计摘要，调用大模型生成字段语义分析与标准化表结构。
   - 问答阶段将用户问题、表结构、样例数据、最近 10 轮历史消息组装为提示词。
   - 大模型输出分析文本和 `<api-call>` 格式的 SQL。
   - 后端提取 SQL，通过 DuckDB 执行，向前端 SSE 推送文本、SQL、图表/表格数据和完成事件。

2. **ReAct Agent 模式**
   - 用户上传文件后，后端创建 ReAct 引擎。
   - 每一轮由大模型输出 `Thought / Action / Action Input`。
   - 后端解析 Action，调用内置工具执行初始统计、Python 代码解释器、文件摘要读取或终止任务。
   - 每轮观察结果写入短期记忆，并结合上下文预算管理继续下一轮。
   - 前端按步骤卡片展示推理、工具调用、输出和最终结论。

### 3.4 后端入口与路由注册

**入口文件**：`backend/main.py`

主要逻辑：

1. 创建 `FastAPI(title="Chat Excel", version="1.0.0", debug=DEBUG)`。
2. 注册 CORS 中间件，当前允许所有来源、方法和请求头。
3. 注册路由：
   - `upload_router` → `/api`
   - `chat_excel_router` → `/api/chat`
   - `chat_react_router` → `/api/chat`
   - `conversation_router` → `/api`
   - `excel_preview_router` → `/api`
   - `sandbox_router` → `/api/sandbox`
4. 启动事件中调用 `init_db()` 初始化 SQLite 表。
5. 提供 `GET /api/health` 健康检查。
6. 直接运行 `main.py` 时调用 `uvicorn.run("main:app", host=HOST, port=PORT, reload=DEBUG)`。

#### 3.5.5 Docker 启动（推荐：跨平台、免装环境）

这是最推荐的方式，尤其适合需要把项目交给同事（Mac / Windows / Linux）运行的场景。用 Docker 把整个环境（Python、Node、依赖）打包进容器，对方只需安装 Docker，一条命令就能跑起来，不需要在本机装 Python、Node 或任何依赖。

**原理**：项目已提供三个文件——`backend/Dockerfile`（后端镜像）、`frontend/Dockerfile`（前端镜像，多阶段构建：Node 编译 → Nginx 提供静态文件 + 反向代理）、`docker-compose.yml`（编排前后端两个容器）。

##### a. 安装 Docker Desktop

- **Windows**：下载 [Docker Desktop for Windows](https://www.docker.com/products/docker-desktop/)，安装时需开启 WSL2（安装器会自动处理），装完启动即可。
- **Mac**：下载 [Docker Desktop for Mac](https://www.docker.com/products/docker-desktop/)，按芯片选 Apple Silicon（M 系列）或 Intel 版本。
- **Linux**：安装 `docker` 和 `docker-compose-plugin`。

安装完成后，在终端执行 `docker --version`，能看到版本号即说明安装成功。

##### b. 配置国内镜像加速器（首次必做）

国内网络直接拉取 Docker Hub 镜像会超时失败，必须先配置国内镜像加速源。

编辑 Docker 配置文件：
- **Windows / Mac**：`~/.docker/daemon.json`（也可在 Docker Desktop 界面 → Settings → Docker Engine 中编辑）
- **Linux**：`/etc/docker/daemon.json`

在配置中加入 `registry-mirrors` 字段：

```json
{
  "registry-mirrors": [
    "https://docker.m.daocloud.io",
    "https://docker.1panel.live",
    "https://hub.rat.dev"
  ]
}
```

保存后重启 Docker 引擎（Windows / Mac 点击 Docker Desktop 界面的 Restart，Linux 执行 `sudo systemctl restart docker`）。

验证是否生效：

```bash
docker info | grep -A 5 "Registry Mirrors"
```

能看到配置的镜像地址即说明生效。

##### c. 一键启动

在项目根目录（`ChatExcel`）执行：

```bash
docker compose up -d --build
```

- `--build`：首次运行或代码变更后构建镜像
- `-d`：后台运行

首次构建需要拉取基础镜像（Python、Node、Nginx）并安装依赖，耗时几分钟；之后启动为秒级。

启动成功后访问：
- 前端：`http://localhost:3030`
- 后端接口文档：`http://localhost:7396/docs`
- 健康检查：`http://localhost:7396/api/health` 返回 `{"status":"ok"}`

## 4. ChatExcel 功能流程

ChatExcel 是项目的核心功能，用户上传 Excel/CSV 文件后，通过自然语言提问，后端自动学习表结构、生成 DuckDB SQL 并执行，以 SSE 流式返回分析文本、SQL、图表数据。

### 4.1 上传流程

**接口**：`POST /api/upload`　　**入口**：`backend/app/api/upload.py` → `upload_file()`

用户上传 Excel/CSV 文件，后端校验后按 `{user_no}/{chat_mode}/{conv_uid}` 三级目录保存到磁盘，同时将每个 Sheet/CSV 落入会话级 DuckDB 临时表，并将文件信息追加到 SQLite conversations 表。如果是 chat_excel 模式，还会创建或追加 ChatExcelEngine 引擎实例到内存缓存。

```text
POST /api/upload → upload_file()        ← backend/app/api/upload.py
  ├→ 校验扩展名（不在 ALLOWED_EXTENSIONS 中则 400）
  ├→ 校验大小（超过 MAX_UPLOAD_SIZE_MB 则 400）
  ├→ _resolve_upload_dir()              ← backend/app/api/upload.py
  │    └→ 生成目录 UPLOAD_DIR/{user_no}/{chat_mode}/{conv_uid}/
  ├→ 生成 file_id，保存文件到磁盘
  └→ 关联会话（chat_mode == "chat_excel" 且有 conv_uid 时）
       ├→ update_conversation()         ← backend/app/dal/conversation.py → 追加文件路径到 SQLite conversations 表
       └→ 创建或追加 ChatExcelEngine     ← backend/app/services/chat_excel/engine.py
            └→ ExcelReader.add_file()   ← backend/app/services/chat_excel/reader.py → 每个 Sheet/CSV 落为 DuckDB temp_* 临时表
```

**补充说明**：

- `_resolve_upload_dir()`：按 `user_no/chat_mode/conv_uid` 三级目录组织，删除会话时可按 `conv_uid` 整体删除。
- `ExcelReader.add_file()`：CSV 落一张临时表，Excel 每个 Sheet 各落一张临时表，表名格式 `temp_{file_key}_{sheet_key}`。
- 返回示例：`{file_id, file_path, file_name, chat_mode, user_no, conv_uid}`。

### 4.2 对话流程

**接口**：`POST /api/chat/excel`　　**入口**：`backend/app/api/chat_excel.py` → `chat_excel()`

用户提问后，路由层根据是否有文件/引擎决定走纯 LLM 对话还是 ChatExcelEngine 流程。进入引擎后，首次上传会触发学习阶段：并发调用 LLM 分析每张表的字段语义，生成 `data_analysis_*` 标准化表。学习完成后进入问答阶段：从 SQLite 加载最近 10 轮历史，组装表结构+样例+历史+当前问题作为提示词，流式调用 LLM 生成回答，解析 `<api-call>` 标签提取 SQL，在 DuckDB 执行查询，通过 SSE 推送文本、SQL、图表数据给前端。

```text
POST /api/chat/excel → chat_excel()     ← backend/app/api/chat_excel.py
  ├→ 取 conv_uid（为空则生成 UUID）
  ├→ 无引擎但有输入 → 尝试从 DuckDB 文件恢复引擎
  │
  ├→ 无文件 + 无引擎 + 有输入
  │    └→ simple_chat_stream()          ← backend/app/api/chat_excel.py → 直接调用 LLM，纯文本 SSE 返回
  │
  └→ 有文件或有引擎 → engine.chat_stream()  ← backend/app/services/chat_excel/engine.py
       │
       ├─【学习阶段】首次上传或新增文件时触发
       │    推送 SSE: {type: "learning"}
       │    ↓
       │    learn() → _learn_table()    ← backend/app/services/chat_excel/engine.py（并发学习每张表）
       │      ├→ 读取临时表 DDL + 前 5 行样例 + SUMMARIZE 统计
       │      ├→ build_learning_messages()  ← backend/app/prompts/chat_excel_learning.py → 组装学习提示词
       │      ├→ chat_completion_full()    ← backend/app/llm/client.py → LLM 非流式调用，返回 JSON
       │      ├→ parse_learning_response() ← backend/app/prompts/chat_excel_learning.py → 解析字段分析
       │      ├→ 有 column_analysis → reader.transform_table()  ← backend/app/services/chat_excel/reader.py → 创建 data_analysis_* 表 + 注释
       │      └→ 无 column_analysis → _fallback_copy_table()    ← backend/app/services/chat_excel/engine.py → 直接复制原始表
       │    ↓
       │    推送 SSE: {type: "learning_result"} + {type: "learning_done"}
       │
       └─【问答阶段】用户有输入时
            _load_chat_history()        ← backend/app/services/chat_excel/engine.py → 从 SQLite 加载最近 10 轮历史消息
            ↓
            获取表结构信息
              ├→ reader.get_all_create_table_sql()  ← backend/app/services/chat_excel/reader.py → 所有已转化表的 DDL
              └→ reader.get_all_sample_data()       ← backend/app/services/chat_excel/reader.py → 所有表前 5 行样例
            ↓
            build_analyze_messages()    ← backend/app/prompts/chat_excel_analyze.py
              └→ 组装 [system(表结构+样例+约束+格式)] + [最近10轮历史] + [当前用户问题]
            ↓
            chat_completion_stream()    ← backend/app/llm/client.py → LLM 流式生成
            ↓
            逐 chunk 处理
              ├→ 累积 full_text
              ├→ <api-call> 之前的文本 → 推送 SSE: {type: "text", content: delta}
              ├→ <api-call> 完整闭合 → parse_api_call()  ← backend/app/services/chat_excel/engine.py → 提取 chart_type 和 sql
              │    ├→ 推送 SSE: {type: "sql", content: sql}
              │    ├→ reader.get_df_by_sql()  ← backend/app/services/chat_excel/reader.py → DuckDB 执行查询
              │    └→ 推送 SSE: {type: "chart", chart_type, sql, data, chart_view}
              ├→ <api-call> 开始但未闭合（兜底）→ 正则提取 <sql>...</sql> → 同上执行并推送
              └→ 无 <api-call>（纯文本回答）→ 推送 SSE: {type: "text"}
            ↓
            推送 SSE: {type: "done"}
```

### 4.4 多轮对话与记忆机制

#### 历史消息加载

**位置**：`backend/app/services/chat_excel/engine.py` → `_load_chat_history()`

- 从 SQLite `messages` 表读取当前会话所有消息。
- `role=human` 映射为 `{"role": "user", "content": ...}`。
- `role=ai` 映射为 `{"role": "assistant", "content": ...}`，优先使用 `metadata.view_message`（含图表占位）。
- 截取最近 **10 轮**（20 条消息），由常量 `CHAT_EXCEL_HISTORY_ROUNDS = 10` 控制。

#### 消息保存

**位置**：`backend/app/services/chat_excel/engine.py` → `_save_round()`

- 用户消息：`role=human, content=用户输入`。
- AI 消息：`role=ai, content=AI 可见文本, metadata={"view_message": 完整展示内容}`。
- `metadata` 中的 `view_message` 包含分析文本 + `<chart-view>` 图表占位，用于历史回放时还原展示。

> **注意**：当前代码中 `_save_round()` 已定义但未在 `chat_stream()` 中调用，消息保存由前端通过 `POST /api/messages` 单独完成。

#### 提示词组装

**位置**：`backend/app/prompts/chat_excel_analyze.py` → `build_analyze_messages()`

组装结构：

```text
[system message]  ← 系统角色 + 模板（表结构 + 样例 + 约束 + 输出格式）
[chat_history]    ← 最近 10 轮历史消息（user/assistant 交替）
[user message]    ← 当前用户问题
```

- 系统提示词包含：采样数据、表 DDL、可用表名、DuckDB 语法规则、输出格式约束。
- 历史消息直接插入，不做压缩或摘要。
- 当前用户问题同时出现在 system 模板的 `{user_input}` 占位符和最后的 user message 中。

#### 记忆机制总结

| 维度 | 实现 | 限制 |
|---|---|---|
| 历史轮数 | 最近 10 轮（20 条消息） | 由 `CHAT_EXCEL_HISTORY_ROUNDS` 常量控制 |
| 历史来源 | SQLite `messages` 表 | 每次对话从数据库重新加载 |
| 历史压缩 | 无 | 不做摘要或截断，直接拼接 |
| 上下文管理 | 无 token 预算控制 | 长对话可能超出模型上下文窗口 |
| 跨会话隔离 | 按 `conv_uid` 隔离 | 不同会话的历史互不影响 |
| 引擎缓存 | 内存中 `_engines` 字典 | 服务重启后需从 DuckDB 文件恢复 |

### 4.5 提示词模块

ChatExcel 有两个提示词模块，分别用于学习阶段和分析阶段。

#### 学习提示词

**位置**：`backend/app/prompts/chat_excel_learning.py` → `build_learning_messages()`

**作用**：让 LLM 分析上传数据的字段语义，返回标准 JSON，用于创建 `data_analysis_*` 标准化表。

**组装内容**：

```text
[system message]
  ├→ 角色设定："你是一个数据分析专家"
  ├→ 采样数据：{data_example}          ← 表前 5 行 JSON
  ├→ 摘要信息：{table_summary}         ← DuckDB SUMMARIZE 结果
  ├→ 表结构信息：{table_schema}        ← CREATE TABLE DDL
  ├→ 分析要求：
  │    - 提取字段列名、数据类型、含义、格式
  │    - 列名标准化转换规则（中文→英文、大写→小写、空格→下划线等）
  │    - 时间类型需给出格式（如 yyyy-MM-dd）
  │    - 提供不同维度的分析思路
  └→ 输出格式：JSON 模板
       {
         "data_analysis": "数据内容分析总结",
         "column_analysis": [{"old_column_name", "new_column_name", "column_description"}],
         "analysis_program": ["1.分析方案1", "2.分析方案2"]
       }

[user message]
  └→ "请分析给你的数据"
```

**LLM 返回解析**：`parse_learning_response()` 清理 markdown 代码块包裹后 `json.loads()`，提取 `data_analysis`、`column_analysis`、`analysis_program` 三个字段。解析失败时降级返回空 `column_analysis`，触发 `_fallback_copy_table()`。

**前端展示生成**：`build_learning_view_message()` 将解析结果转为 Markdown，包含"数据摘要"、"数据结构"（列名映射）、"分析建议"三部分。

#### 分析提示词

**位置**：`backend/app/prompts/chat_excel_analyze.py` → `build_analyze_messages()`

**作用**：让 LLM 根据用户问题生成 DuckDB SQL，以 `<api-call>` 格式返回，用于数据查询和图表展示。

**组装内容**：

```text
[system message]
  ├→ 角色设定："你是一个数据分析专家！"
  ├→ 采样数据：{data_example}          ← 所有表前 5 行 JSON（含 file_name/sheet_name/table_name/columns/rows）
  ├→ 表结构信息：{table_schema}        ← 所有已转化表的 CREATE TABLE DDL
  ├→ 可用表名：{table_names}           ← 逗号分隔的表名列表
  ├→ DuckDB 语法规则：
  │    - GROUP BY 非聚合列必须出现在 GROUP BY 中
  │    - 多层 CTE 列引用一致性
  │    - 时间戳使用 to_timestamp() 而非直接 CAST
  ├→ 约束条件（12 条）：
  │    - 必须使用 DuckDB SQL 分析
  │    - 从可用展示方式中选择最优类型
  │    - 只能使用给定表名，不能臆造
  │    - 多表分析需说明关联依据
  │    - 数据相关问题必须生成 <api-call>，禁止只输出自然语言结论
  │    - 非数据问题直接说明无法回答，不生成 <api-call>
  ├→ 输出格式：
  │    [分析思路]
  │    1. 核心查询目标
  │    2. 选择哪些表、哪些字段及依据
  │    3. 聚合、筛选、排序、关联方式
  │    4. 结果展示方式
  │    <api-call><name>[展示方式]</name><args><sql>[SQL]</sql></args></api-call>
  ├→ 硬性输出要求：
  │    - 数据问题必须以 [分析思路] 开头，按 1-4 编号
  │    - 必须包含完整 <api-call> 标签
  │    - <sql> 必须是可直接执行的 SELECT 语句
  │    - 禁止直接输出统计结论（如"共有5人"）
  │    - 不要输出多个 <api-call>
  ├→ 可用展示方式：response_line_chart / response_pie_chart / response_table /
  │    response_scatter_chart / response_bubble_chart / response_donut_chart /
  │    response_area_chart / response_heatmap / response_vector_chart
  ├→ 示例：完整的 [分析思路] + <api-call> 示例
  └→ 用户问题：{user_input}            ← 当前用户问题（同时出现在 system 模板和最后的 user message 中）

[chat_history]                          ← 最近 10 轮历史消息（user/assistant 交替）

[user message]
  └→ 当前用户问题（与 system 模板中的 {user_input} 内容相同）
```

**两个提示词的对比**：

| 维度 | 学习提示词 | 分析提示词 |
|---|---|---|
| 调用方式 | 非流式（`chat_completion_full`） | 流式（`chat_completion_stream`） |
| 输出格式 | JSON | [分析思路] + `<api-call>` 标签 |
| 是否含历史 | 否 | 是（最近 10 轮） |
| 核心目标 | 字段语义分析 + 列名标准化 | 生成可执行的 DuckDB SQL |
| 失败降级 | 空列映射 → 直接复制原始表 | 正则兜底提取 `<sql>` |

## 5. ReAct Agent 功能流程

### 5.1 整体定位

ReAct Agent 是项目的第二条分析链路，面向**多表多 Sheet、需要多步推理和 Python 代码执行的复杂分析场景**。与 ChatExcel SQL 模式的区别：

| 维度 | ChatExcel SQL 模式 | ReAct Agent 模式 |
|---|---|---|
| 分析方式 | LLM 生成 SQL → DuckDB 执行 | LLM 逐步推理 → 调用工具执行代码 |
| 文件支持 | 单文件，按 Sheet 入 DuckDB | 多文件，直接用 pandas 读取 |
| 执行引擎 | DuckDB | Python 子进程（code_interpreter） |
| 多步推理 | 单轮（学习 + 问答） | 多轮 ReAct 循环（最多 30 轮） |
| 中间过程 | 对用户不可见 | 前端实时展示每一步 Thought / Action / Observation |
| 上下文管理 | 最近 10 轮历史消息 | 短期记忆（5 轮）+ 4 层压缩 |

### 5.2 接口入口

**接口声明**：`POST /api/chat/react-agent`（SSE 流式响应）
**入口文件**：`backend/app/api/chat_react.py` → `chat_react_agent()`
**请求体**：`{ user_input, conv_uid, file_path, file_name, file_paths, file_names, skill_name }`

### 5.3 三条分支

接口收到请求后，根据 **是否已有引擎** 和 **是否上传了文件** 走三条分支：

```text
chat_react_agent()
  │
  ├→ has_engine = conv_uid in _engines ?    ← _engines 是内存字典，非数据库
  ├→ has_file = bool(file_paths)
  │
  ├→【分支1】not has_file and not has_engine
  │    → 纯 LLM 对话（不创建引擎）
  │    → 单轮流式输出，不走 ReAct 循环
  │
  ├→【分支2】has_file and conv_uid not in _engines
  │    → 创建新 ReactEngine，放入 _engines[conv_uid]
  │    → 首次提问时才创建，上传文件时不会创建
  │
  └→【分支3】conv_uid in _engines
       → 复用已有引擎
       → 如果带了新文件，追加到 engine.file_paths
```

> **易误解点**：`conv_uid` 在上传第一个文件时就生成了（存在数据库），但 `_engines` 是内存字典，只有第一次提问时才创建 `ReactEngine` 实例并放入。上传文件 ≠ 创建引擎。

### 5.4 ReAct 循环主流程

**入口函数**：`ReactEngine.run_stream()`（`backend/app/services/react_agent/engine.py`）

```text
run_stream(user_input)
  │
  ├→ memory.clear()                    ← 清空上一问的记忆，避免跨问题污染
  │
  └→ while retry_count < max_retry (30):
       │
       ├─ 1. 推送 step.start 事件（SSE）
       │
       ├─ 2. _load_thinking_messages(current_input)
       │    ├→ 组装 system prompt（含工具描述、文件路径、历史摘要、task_progress）
       │    ├→ 读取短期记忆 memory.read_as_messages()（最近 5 轮 react 自身的执行步骤的记忆）
       │    ├→ 追加当前输入 user message
       │    ├→ ctx_mgr.manage_context()（L1/L2/L3 压缩检查）
       │    └→ 推送 context.status 事件（SSE）
       │
       ├─ 3. 调 LLM（流式）
       │    ├→ chat_completion_stream(messages)
       │    ├→ 实时提取 Thought 推送 step.chunk 事件
       │    ├→ 如果 LLM 返回上下文溢出错误 → L4 紧急压缩 → 重试1次
       │    └→ 累积完整 llm_output
       │
       ├─ 4. parse_react_output(llm_output)
       │    ├→ 用括号计数法提取第一个完整 JSON（不贪婪匹配）
       │    ├→ 解析出 thought / action / action_input
       │    └→ 推送 step.meta 事件（SSE）
       │
       ├─ 5. 如果 action == "terminate"
       │    ├→ 提取 final_content = action_input["result"]
       │    ├→ memory.write(terminate 片段)
       │    ├→ 推送 step.done + final + done 事件
       │    └→ return（结束循环）
       │
       ├─ 6. 如果无有效 action
       │    ├→ 推送重试提示
       │    └→ current_input = 重试提示, continue
       │
       ├─ 7. 执行工具 execute_tool(action, action_input)
       │    ├→ 运行 Python 代码 / 读取文件 / 渲染 HTML 等
       │    ├→ 推送 step.chunk 事件（工具输出）
       │    └→ 推送 step.done 事件
       │
       ├─ 8. memory.write(本轮片段)
       │    ├→ question = current_input
       │    ├→ thought / action / action_input / observation
       │    └→ 超过 5 条自动丢弃最旧的
       │
       └─ 9. current_input = "Observation: {observation_text}"
            → 继续下一轮循环
```

### 5.5 消息组装详解

每一轮 ReAct 循环都会重新组装发给 LLM 的 messages 列表：

```text
messages = [
  {role: "system",  content: 系统提示词},
  {role: "user",    content: 记忆片段1（第1轮的 Thought+Action+Observation）},
  {role: "user",    content: 记忆片段2},
  {role: "user",    content: 记忆片段3},
  {role: "user",    content: 记忆片段4},
  {role: "user",    content: 记忆片段5},
  {role: "user",    content: 当前输入（第1轮是用户问题，后续轮是 Observation）},
]
```

> **易误解点**：记忆不是拼到 system prompt 里，而是作为独立的 `user` 消息插入到 system 和当前输入之间。每轮循环都会重新读 memory、重新组装。

### 5.6 短期记忆机制

**位置**：`backend/app/core/memory.py` → `ShortTermMemory`

| 属性 | 说明 |
|---|---|
| `buffer_size` | 来自 `.env` 的 `SHORT_TERM_MEMORY_BUFFER_SIZE=5` |
| 存储结构 | 内存列表 `_buffer: List[MemoryFragment]` |
| 写入 | `write(fragment)` → append，超过 5 条 pop(0) |
| 读取 | `read_as_messages()` → 每条片段转为 `{"role": "user", "content": "Thought: ...\nAction: ...\nObservation: ..."}` |
| 清空 | 每个新用户问题开始时 `memory.clear()` |

**MemoryFragment 结构**：

```python
class MemoryFragment:
    question: str      # 当前轮输入（第1轮是用户问题，后续是 Observation）
    thought: str       # LLM 的思考
    action: str        # 选择的工具名
    action_input: str  # 工具参数（JSON 字符串）
    observation: str   # 工具返回结果
```

> **易误解点**：memory 的 5 轮是**当前问题 ReAct 循环内的中间步骤**，不是跨问题的历史对话。每个新问题开始时都会清空。跨问题的上下文由系统提示词里的 `conversation_history` 提供。

### 5.7 系统提示词中的上下文层次

系统提示词（system message）里包含**两个层次的上下文**，与短期记忆互补：

| 上下文层次 | 来源 | 内容 | 粒度 |
|---|---|---|---|
| `conversation_history` | 从数据库读取历史问答 | `用户问: xxx → 结论: xxx` | 粗粒度，只有问题和最终结论，截断 500 字 |
| `task_progress` | 当前问题 ReAct 循环累积 | `1. 完成: xxx\n2. 完成: xxx` | 中粒度，每轮的完成摘要，截断 100 字 |
| 短期记忆 memory | 当前问题 ReAct 循环中间步骤 | `Thought + Action + Observation` | 细粒度，完整步骤细节 |

三者不重叠：

- `conversation_history` 管"之前问过什么"（跨问题）
- `task_progress` 管"当前问题做到哪了"（全局进度）
- `memory` 管"当前问题最近几步的细节"（局部步骤）

### 5.8 4 层上下文压缩

**位置**：`backend/app/core/context.py` → `ContextManager`

每一轮 ReAct 循环组装完 messages 后，都会调用 `manage_context()` 检查 token 占用率：

```text
manage_context(messages, memory)
  │
  ├→ state = get_context_state(messages)
  │   用 tiktoken 本地统计 token 数（不调大模型）
  │   budget = CONTEXT_MAX_TOKENS(120000) - CONTEXT_RESERVED_TOKENS(4096)
  │
  ├→ state == "normal" (< 70%)  → 不压缩，直接返回
  │
  ├→ state in ("warning","error","overflow") (>= 70%)
  │   ├→ L1: 截断旧 Observation 到 500 字符
  │   ├→ L2: 只保留最近 3 轮（6 条消息）
  │   └→ L3 (仅 error/overflow >= 90%): 调 LLM 对旧消息生成摘要
  │
  └→ L4 (reactive_compact): LLM 返回溢出错误时触发
      只保留 system + 最近 1 轮（2 条消息）
```

| 层级 | 触发条件 | 做什么 | 是否调 LLM |
|---|---|---|---|
| L1 | token >= 70% | 截断旧 Observation 到 500 字符 | 否 |
| L2 | token >= 70% | 丢弃旧轮次，只保留最近 3 轮 | 否 |
| L3 | token >= 90% | 调 LLM 把旧消息压缩成摘要 | 是 |
| L4 | LLM 返回溢出错误 | 紧急模式，只保留 system + 最近 1 轮 | 否 |

> **token 统计方式**：用 `tiktoken`（OpenAI BPE 分词器）本地计算，不调大模型。降级方案是 `len(text) / 3`。

### 5.9 ReAct 输出解析

**位置**：`engine.py` → `parse_react_output(text)`

LLM 输出格式：

```text
Thought: 分析当前任务状态...
Action Intention: 这一步要做什么
Action Reason: 为什么需要这一步
Action: code_interpreter
Action Input: {"code": "import pandas as pd\n..."}
```

解析逻辑：

1. 正则提取 `Thought`、`Action Intention`、`Action Reason`、`Action`
2. `Action Input` 用**括号计数法**提取第一个完整 JSON 对象（不贪婪匹配到最后一个 `}`）

> **易误解点**：旧版用贪婪正则 `\{.*\}` 匹配，当模型一次输出多轮（自带 Observation + 下一轮）时会把整个文本都包进去，导致 JSON 解析失败，fallback 成 `{"raw": 整段文本}`，工具拿不到 `code` 参数返回 "No code provided"。已修复为括号计数法。

### 5.10 内置工具

**位置**：`backend/app/services/react_agent/tools.py` → `execute_tool()`

| 工具名 | 参数 | 用途 |
|---|---|---|
| `execute_analysis` | `{"input_file": "路径"}` | 快速分析文件概况（shape/columns/dtypes/head5），读取所有非空 Sheet |
| `code_interpreter` | `{"code": "Python代码"}` | 执行任意 Python 代码，预注入 pandas/numpy/matplotlib/FILE_PATH/DATAFRAMES |
| `html_interpreter` | `{"html": "...", "title": "..."}` | 渲染 HTML 报告到右侧面板 |
| `load_file` | `{"file_path": "路径"}` | 加载上传文件信息 |
| `todowrite` | `{"todos": [...]}` | 创建/更新任务计划，前端实时展示进度 |
| `terminate` | `{"result": "最终答案"}` | 结束任务，返回最终结论 |
| `select_skill` | `{"query": "..."}` | 根据用户查询选择最相关的 skill |
| `load_skill` | `{"skill_name": "...", "file_path": "..."}` | 加载 skill 内容 |
| `get_skill_resource` | `{"skill_name": "...", "resource_path": "..."}` | 读取 skill 参考文档 |
| `execute_skill_script` | `{"skill_name": "...", "script_name": "..."}` | 执行 skill 内联脚本 |
| `execute_skill_script_file` | `{"skill_name": "...", "script_file_name": "..."}` | 执行 skill 脚本文件 |

**工具加载机制（重要）**：

本项目**没有动态工具注册机制**（不像 LangChain 的 `@tool` 装饰器自动生成描述）。工具的名称、描述、参数格式是**硬编码在系统提示词模板里**的纯文本，工具执行是 `execute_tool()` 里写死的 `if/elif` 分支。涉及两个位置：

| 阶段 | 位置 | 做了什么 |
|---|---|---|
| **提示词构建** | `backend/app/prompts/react_agent_system.py` 的 `REACT_SYSTEM_TEMPLATE` | 工具描述（名称 + 参数格式 + 使用说明）写死在 `## Available Tools Description` 段落中，随系统提示词一起发给 LLM |
| **工具执行** | `backend/app/services/react_agent/tools.py` 的 `execute_tool()` | LLM 返回 `Action: 工具名` 后，用 `if/elif` 字符串匹配分发到对应函数 |

新增工具需同时改两处：① 模板里加工具描述 ② `execute_tool()` 里加 `elif` 分支。

**code_interpreter 预注入环境**：

```python
import json, os, sys, pandas as pd, numpy as np
FILE_PATH = r"上传文件路径"
DATAFRAMES = load_all_sheets(FILE_PATH)    # 所有非空 Sheet 的字典
df = next(iter(DATAFRAMES.values()))       # 兼容旧代码的默认 df
```

> 每次调用完全独立，变量不跨调用持久化。Excel 文件强制读取所有非空 Sheet（`sheet_name=None`），空 Sheet 自动过滤。

### 5.12 ReAct 系统提示词

**位置**：`backend/app/prompts/react_agent_system.py` → `REACT_SYSTEM_TEMPLATE`

组装结构：

```text
[system message]
  ├→ 角色设定："DB-GPT intelligent assistant"
  ├→ 自主决策原则（6 条）
  │    - 分析需求 → 选工具 → 输出 Thought/Action/Action Input
  │    - 等待 Observation 后再决定下一步
  │    - 完成时调 terminate
  │    - 报告类任务必须调 html_interpreter
  ├→ 任务管理（todowrite 使用规范）
  │    - 3 步以上任务必须用 todowrite
  │    - 每次状态变更都要调用 todowrite
  ├→ 可用 Skills 列表（预加载）
  ├→ Skill 执行规范
  │    - 理解工作流 → 按顺序执行
  │    - 资源使用时机：计算用脚本、理解用参考文档
  ├→ 可用工具描述（11 个工具的参数格式）
  │    - code_interpreter: 变量不持久、Excel 必须读所有 Sheet
  │    - html_interpreter: 报告渲染的唯一方式
  │    - todowrite: 每次传完整列表
  ├→ 文件上下文
  │    - 单文件：路径 + 文件名
  │    - 多文件：文件列表 + 主文件标注
  ├→ ReAct 输出格式约束
  │    Thought / Action Intention / Action Reason / Action / Action Input
  │    Action Intention <= 18 中文或 8 英文
  │    Action Reason <= 30 中文或 12 英文
  ├→ 跨问题历史摘要（conversation_history）
  │    - 用户问: xxx → 结论: xxx
  ├→ 当前问题进度（task_progress）
  │    - 1. 完成: xxx
  │    - 2. 完成: xxx
  └→ 语言要求：中文交流，代码相关内容可英文
```

**记忆机制（重要）**：

ReAct Agent 有三个不同层级的"记忆"机制，作用域和职责各不相同，互补确保 LLM 不会丢失上下文：

| 机制 | 作用域 | 保留内容 | 清空时机 | 代码位置 |
|---|---|---|---|---|
| **ShortTermMemory（短期记忆）** | 同一问题的 ReAct 循环内，最近 5 轮 | 每轮的 Thought + Observation（完整文本） | 每个新问题开始时 `memory.clear()` | `engine.py` 的 `run_stream()` |
| **Task Progress（任务进度）** | 当前问题的所有步骤摘要 | 每步的工具调用摘要（如 `execute_analysis(...)`, `code_interpreter(...)`），不丢不挤 | 不清空，问题内持续累加 | `engine.py` 的 `self.task_progress` |
| **Conversation History（跨问题历史）** | 跨问题的历史结论 | 之前问题的"用户问 + 结论"摘要 | 不清空，从 DB 的 `finalContent` 读取 | `engine.py` 的 `_build_conversation_history()` |

**ShortTermMemory 与 Task Progress 的互补关系**：

ShortTermMemory 只保留最近 5 轮的详细内容（Thought、代码、输出），超过 5 轮后早期步骤会被挤出。Task Progress 累积记录所有步骤的摘要，不丢失。两者互补：假设一个问题跑了 10 轮 ReAct，到第 10 轮时：

- ShortTermMemory 里只剩第 6-10 轮的**详细**内容（第 1-5 轮被挤出）
- Task Progress 里有第 1-10 轮的**摘要**（每步做了什么）

LLM 既能通过 Task Progress 掌握全局进展（知道前几步做过什么），又能通过 ShortTermMemory 看到最近 5 轮的细节（具体代码和输出）。

**Task Progress 与 ShortTermMemory 的信息粒度区别（为什么不重复）**：

两者信息量差异很大，不是重复关系——Task Progress 是**目录**，ShortTermMemory 是**正文**：

| 内容 | Task Progress | ShortTermMemory |
|---|---|---|
| Thought（LLM 的推理过程） | ❌ 没有 | ✅ 完整 |
| Action + Action Input | ✅ 截断到 50 字符 | ✅ 完整 |
| Observation（工具返回的结果） | ❌ 没有 | ✅ 完整（代码、表格、JSON、错误信息） |

Task Progress 的存储逻辑（`engine.py`）：

```python
# 只存工具名 + 截断到 50 字符的参数
action_desc = f"{parsed['action']}({json.dumps(parsed['action_input'], ensure_ascii=False)[:50]})"
self.task_progress.append(action_desc)
```

举例：第 3 轮用 code_interpreter 跑了 30 行 Python 代码、返回了 50 行统计表：

- Task Progress 里只有：`code_interpreter({"code": "import pandas as pd\n\n# 加载)`
- ShortTermMemory 里有：完整的 Thought + 完整的 30 行代码 + 完整的 50 行结果表

如果只剩 Task Progress，LLM 只知道"第 3 步用了 code_interpreter"，但**不知道执行了什么、结果是什么**。到了第 8 轮如果想引用第 3 步的结果，没有 ShortTermMemory 就得重新跑一遍代码。

**为什么跨问题要清空 memory？**

每个新问题开始时 `memory.clear()` 清空上一问的 ReAct 轨迹，避免上一问的 `terminate(result=xxx)` 最终答案污染当前问题，导致模型复用旧答案而不重新分析。跨问题的上下文由 Conversation History 从 DB 提取精简摘要补偿。

### 5.13 数据库与存储

ReAct 模式涉及的存储：

| 存储 | 位置 | 用途 |
|---|---|---|
| SQLite `chat_messages` 表 | `storage/db/chat_excel.db` | 保存每轮问答（user_input + final_content + steps） |
| SQLite `conversations` 表 | 同上 | 保存会话元信息（conv_uid, chat_mode, file_path 等） |
| 上传文件 | `storage/uploads/default/react_agent/{conv_uid}/` | 用户上传的原始 Excel/CSV 文件 |
| 代码执行临时目录 | `storage/tmp/{conv_uid}/` | code_interpreter 的 `_run.py` 临时脚本，执行后自动删除 |
| 生成图片 | `storage/static/images/` | code_interpreter 生成的图表，拷贝后通过 `/images/` 静态服务访问 |
| 会话日志 | `logs/user/react_agent/{conv_uid}.log` | 每个会话独立日志文件，记录完整 ReAct 过程 |
| 引擎实例 | 内存 `_engines` 字典 | ReactEngine 实例，持有文件路径、memory、todo 等，进程重启后丢失 |

> ReAct 模式不使用 DuckDB，不使用 `react_agent_state/` 目录（已废弃删除）。引擎状态只在内存中，不持久化到磁盘。

### 5.14 LLM 调用与重试

**位置**：`backend/app/llm/client.py`

| 配置项 | 来源 | 默认值 | 说明 |
|---|---|---|---|
| `LLM_RETRY_COUNT` | `.env` | 3 | 最大尝试次数（含首次），失败自动切换下一个模型 |
| `LLM_NON_STREAM_TIMEOUT` | `.env` | 60 | 非流式调用整体超时（秒） |
| `LLM_STREAM_FIRST_CHUNK_TIMEOUT` | `.env` | 10 | 流式调用首个内容 chunk 超时（秒） |
| 模型配置 | `llm_config.py` | 代码内配置 | 多模型列表，重试时按顺序轮换 |

重试逻辑：

- 每次重试自动切换到下一个 provider（轮询）
- 触发重试的条件：连接失败、超时、限流、5xx、流式首个 chunk 超时
- 流式调用如果已输出首个 chunk 后中断，不重试（避免重复输出）

### 5.15 前端交互流程

**位置**：`frontend/src/components/ChatPanel.tsx` + `frontend/src/hooks/useReactAgent.ts`

```text
用户提问
  │
  ├→ POST /api/chat/react-agent (SSE)
  │
  ├→ 收到 step.start → 创建步骤卡片，状态 running
  ├→ 收到 step.meta → 填充 thought/action/actionInput
  ├→ 收到 step.chunk → 实时追加输出（thought/text/table/image/code）
  ├→ 收到 context.status → 更新顶部上下文使用率条
  ├→ 收到 plan.update → 更新任务计划列表
  ├→ 收到 step.done → 标记步骤完成
  ├→ 收到 final → 填充结论区域（ReactMarkdown 渲染）
  ├→ 收到 error → 显示错误，标记步骤失败
  └→ 收到 done → 停止 loading
```

> 前端对 `finalContent` 做了清洗：如果后端返回 `{"result": "..."}` 格式的 JSON 字符串，自动提取 `result` 字段后再交给 ReactMarkdown 渲染。
