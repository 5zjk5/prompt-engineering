# Project Analyzer — 代码仓库梳理

## 概述

此技能用于系统性梳理代码仓库，生成结构化的项目文档。通过分阶段探索代码仓——从宏观架构到微观接口细节——最终输出一份按模板组织的 Markdown 文件，帮助用户快速理解项目的定位、架构、业务流程和接口逻辑。

## 核心工作流

当用户触发此技能时，按以下流程执行：

### 阶段一：读取梳理模板

1. 读取 `references/project-analysis-template.md`，了解输出文档的结构和各节要求。
2. 模板定义了以下核心章节（根据项目实际情况取舍）：
   - 项目一句话定位
   - 解决什么问题（痛点 + 处理思路）
   - 技术架构（前端/后端框架、关键依赖、启动方式）
   - 端到端业务流程（接口顺序 / 独立接口简介）
   - 后端接口详细流程（逐接口：定义、入参、返回值、代码执行步骤）
   - 根据项目情况追加的其他章节

**关键原则**：模板是参考框架而非强制约束。根据项目实际情况调整章节结构——简单项目可简化，复杂项目可扩展。模板中的【】占位符为填写指引，最终输出应替换为实际内容。

### 阶段二：宏观探索 — 理解项目全貌

目标：建立对项目整体结构的认知，不深入代码细节。

1. **识别项目类型**：查看根目录文件（README.md、package.json、pyproject.toml、requirements.txt、Makefile、docker-compose.yml 等），判断项目语言、框架、前后端结构。
2. **梳理目录结构**：列出顶层目录和关键子目录，理解代码组织方式（如 monorepo、前后端分离、模块化等）。
3. **识别入口文件**：找到服务启动入口（main.py、app.py、index.ts、server.js 等），了解服务如何启动。
4. **识别配置文件**：查看 .env、config/、settings 等，了解环境变量和配置结构。
5. **识别路由/接口定义**：找到 API 路由注册位置（routes/、api/、controllers/ 等），建立接口清单。
6. **识别数据模型**：找到 models/、schemas/、entities/ 等，了解核心数据结构。

**产出**：对项目的整体架构、技术栈、目录组织有清晰认知，为后续深入梳理奠定基础。

### 阶段三：微观深入 — 逐模块梳理

目标：深入代码逻辑，梳理每个模块/接口的具体流程。

1. **接口梳理**（按路由定义逐个分析）：
   - 路由定义（HTTP 方法、路径、处理函数）
   - 请求/响应模型（Pydantic model、TypeScript interface、JSON schema 等）
   - 服务层逻辑：调用链、条件分支、异常处理
   - 逐步骤描述代码执行流程（参考模板 5.1 节的详细程度）
   - 依赖的外部服务（数据库、缓存、消息队列、第三方 API 等）

2. **核心业务逻辑梳理**：
   - 识别核心算法/处理逻辑
   - 理解数据流转：输入 → 处理 → 输出
   - 关键配置常量和业务规则

3. **前后端交互梳理**（如适用）：
   - 前端调用了哪些后端接口
   - 接口调用的先后顺序
   - 数据在前端的处理和展示逻辑

**关键原则**：
- 接口梳理优先按端到端业务流程顺序排列（先梳理主流程，再梳理辅助接口）
- 复杂接口需要逐步骤描述代码执行流程，标注关键函数调用和条件判断
- 简单接口可简化描述，不必强行拆分为多个步骤
- 入参和返回值应列出字段说明表，复杂嵌套结构需逐层展开

### 阶段四：生成输出文档

1. **按模板结构组织内容**：将梳理结果填入模板定义的各章节中。
2. **自定义调整**：
   - 如果项目没有前后端分离，省略前端相关章节
   - 如果接口间有明确先后顺序，使用"接口先后顺序"结构
   - 如果接口相互独立，使用"接口简介"结构
   - 根据项目特点追加必要的章节（如部署说明、数据模型、配置说明等）
3. **输出 Markdown 文件**：将最终文档写入用户指定的路径，或默认放在项目根目录下命名为 `项目梳理.md`。

## 代码探索策略

在阶段二和阶段三中，使用以下策略高效探索代码：

### 探索工具选择

- **目录结构**：优先使用 Glob 工具匹配文件模式（`**/*.py`、`**/routes/*` 等）
- **关键定义搜索**：使用 Grep 工具搜索路由装饰器（`@router`、`@app`）、类定义（`class`）、配置项等
- **文件阅读**：使用 Read 工具逐文件阅读关键源码
- **大规模探索**：当代码仓较大时，优先使用 Agent（Explore 类型）进行并行探索

### 探索顺序建议

1. 入口文件 → 路由注册 → 服务层 → 数据层
2. 主业务流程 → 辅助功能 → 工具函数
3. 核心模块 → 扩展模块 → 测试文件

### 深度控制

- 接口梳理：深入到服务层内部，逐步骤描述执行流程
- 工具函数：仅说明用途和关键参数，不必展开实现
- 第三方库调用：说明调用了什么、传了什么参数，不必展开库内部逻辑
- 配置项：列出关键配置项和默认值

## 输出质量要求

- **准确性**：所有代码路径、函数名、文件位置必须与实际代码一致，不得臆造
- **完整性**：核心接口和业务流程必须全部覆盖，不得遗漏
- **可读性**：技术描述应兼顾深度与可理解性，避免堆砌代码片段
- **实用性**：文档应能让新成员快速理解项目架构和核心流程
- **结构化**：严格按照模板章节组织，使用清晰的标题层级

### references/project-analysis-template.md

项目梳理输出模板。定义了文档的标准结构和各章节的填写指引。在阶段一必须读取此文件，在阶段四按此模板组织输出内容。

模板中的关键章节说明：
- **第1节 项目一句话定位**：用一句话概括项目用途
- **第2节 解决什么问题**：描述痛点 + 项目的处理思路
- **第3节 技术架构**：框架、依赖、启动方式、VSCode 调试配置
- **第4节 端到端业务流程**：接口调用顺序或独立接口简介
- **第5节 后端接口详细流程**：每个接口的定义、入参、返回值、代码执行步骤
- **第8-9节**：根据项目情况追加的扩展章节
