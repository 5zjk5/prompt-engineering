## 代码编写规则

### 1. 坚持最小改动原则

每次修改都应尽量聚焦在当前需求或问题本身，只改动与目标直接相关的代码。已有逻辑、已有结构、无关模块和稳定功能不应被顺手调整，避免引入额外风险。

在处理问题时，应优先定位真正需要变更的位置，而不是扩大修改范围。除非需求明确要求重构或调整架构，否则不要因为个人偏好去改写已有实现。

### 2. 严格保持既有格式规范

不得随意修改已有代码的格式、缩进、换行、排序或风格。即使发现某些代码不符合常见格式规范，只要它与当前需求或问题无关，就不能主动格式化或调整。

例如，Python 代码中某一行较长，但它并不属于当前修改范围，则禁止因为格式原因改动它。保持原有代码格式的稳定性优先级高于一切，避免无关 diff 干扰代码审查，也避免隐藏真实变更。

### 3. 避免过度复杂化

实现需求时，应始终关注代码的可读性、可维护性和复杂度控制。在功能正确、性能满足要求的前提下，优先选择最简单、最直接、最容易理解的实现方式。

不要为了追求抽象、通用性或设计感而引入不必要的层级、工具函数、复杂模式或额外依赖。只有当复杂性能够明确解决当前问题，并且收益大于维护成本时，才允许引入更复杂的方案。

### 4. 以生产环境标准编写代码

编写代码时必须考虑真实生产环境的影响，目标是交付可靠、可维护、可运行的生产代码，而不是仅能演示功能的简单 demo。

实现过程中需要关注边界条件、异常处理、性能影响、并发安全、异步流程、资源释放、数据库连接使用、事务一致性、超时重试、日志输出和安全风险等问题。涉及外部服务、文件系统、网络请求、数据库或缓存时，应特别注意失败场景和资源管理。

任何看似简单的改动，都应评估它在生产流量、异常输入、并发调用和长期运行情况下的表现，避免留下隐患。


### 5. 外部调用必须兜底防挂

调用大模型接口、Embedding 接口、向量数据库、外部 HTTP 服务等不可控依赖时，必须做异常兜底，确保单次调用失败不会导致整个流程崩溃。具体要求：

- 大模型调用（chat / embedding）必须 try-except 包裹，捕获异常后记录日志并返回合理的降级结果，而非直接抛出让上层崩溃。例如：chat 调用失败可返回空字符串或预设兜底内容，embedding 调用失败可返回零向量或跳过当前批次。
- 批量处理场景中，单条数据的调用失败不应中断整个批次的执行。失败的条目应记录错误信息并填充占位结果，保证后续数据继续处理。
- 涉及网络请求、文件读写、数据库操作等 IO 场景，同样需要 try-except 兜底，避免因单点故障导致进程退出。
- 兜底逻辑应打印或记录足够的错误信息（异常类型、消息、相关上下文），便于事后排查，但不能因为日志本身再抛异常。

### 6. 注释必须使用中文，函数必须标注用途

编写代码时，注释和文档字符串必须使用中文。每个函数都必须添加注释，说明该函数的作用，让阅读者无需逐行阅读代码就能快速理解函数的功能。

具体要求：

- **函数注释**：每个函数定义的下方或上方，必须用中文简要说明该函数的功能。Python 函数应使用 docstring（`""" """`），其他语言使用对应的注释语法。说明应包含函数做了什么，必要时补充关键参数和返回值的含义。
- **行内注释**：当代码逻辑不够直观时（如业务规则、特殊处理、边界条件、算法步骤等），必须在关键行或代码块上方添加中文注释，解释"为什么这样做"或"这里在做什么"。显而易见的逻辑不需要加注释，避免注释噪音。
- **注释语言**：所有注释必须使用中文，包括 docstring、行内注释、TODO 标记等，确保团队内阅读无障碍。

### 7. 测试脚本必须使用固定的可配置格式

在编写批量跑数据的测试脚本时，必须保持统一、清晰、可快速调整的代码结构。所有关键参数都应放在脚本开头，使用变量集中保存，便于直接修改和配置，避免在脚本逻辑中散落硬编码。

脚本开头必须使用变量统一定义以下内容：输入数据路径、输出结果路径、大模型 API URL、大模型名称、embedding 模型 API URL、embedding 模型名称，以及其他与运行环境相关的配置项。变量命名应清晰、统一，方便一眼看出含义。其中，输入数据路径和输出结果路径必须使用绝对路径，禁止使用相对路径，避免因脚本运行目录不同导致路径解析错误。

脚本必须包含 `debug` 变量，默认值为 `False`。当 `debug` 为 `True` 时，默认只跑前 10 条数据，用于快速验证脚本是否能正常运行，避免在调试阶段误跑全量数据导致时间、Token 或资源浪费。所有批量逻辑都必须遵守该开关，不允许绕过。

如果脚本涉及大模型调用，必须提供批次调用数量的配置变量（例如 `batch_size`），默认值为 `5`，即每个批次调用 5 条数据。批次大小必须能在脚本开头集中修改，方便根据模型限流、并发能力和数据规模灵活调整，避免在循环逻辑中硬编码批次数量。这里的批次指的是并发，指的是并发数！！！！！

如果测试脚本测试的是 FastAPI 接口，并且后端接口代码与测试代码位于同一目录或可直接导入同一个 `app` 实例，则必须优先使用 FastAPI 提供的 `TestClient` 或 `httpx.AsyncClient` 直接实例化应用进行测试，模拟服务运行过程，不允许为了测试再手动启动外部服务。
